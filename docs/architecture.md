## 2. 核心技术点深度解析

### 2.1 分布式 ID 生成 (Snowflake)

**为什么不用 MySQL 自增 ID？**
- 自增 ID 在分库分表时会冲突
- 暴露业务量（竞品可以通过 ID 推断你的日活）
- 无法在应用层提前生成，必须等数据库返回

**Snowflake 设计：**
```
| 1bit 符号位 | 41bit 时间戳 | 10bit 机器ID | 12bit 序列号 |
|     0       | 毫秒级时间   | datacenter+worker | 4096/ms  |
```
- 支持 69 年不重复
- 每台机器每毫秒 4096 个 ID
- 1024 台机器并行
- ID 天然有序，可做消息排序

### 2.2 Feed 推荐算法

**召回 -> 粗排 -> 精排 -> 重排 四阶段管线：**

1. **召回 (Recall)**：从亿级视频池中快速筛出候选集（~1000）
   - 协同过滤：看了 A 的人也看了 B
   - 内容标签匹配：用户兴趣标签 vs 视频标签
   - 热门池：近 24h 互动率 top
   - 关注流：关注的人发的新视频

2. **粗排 (Pre-rank)**：用轻量模型快速打分（~200）
   - 特征：用户画像 + 视频特征 + 上下文
   - 目标：预估 CTR（点击率）

3. **精排 (Rank)**：复杂模型精确打分（~50）
   - 多目标：完播率 × 0.4 + 点赞率 × 0.3 + 评论率 × 0.2 + 分享率 × 0.1
   - 时间衰减：Wilson Score + 时间衰减因子

4. **重排 (Re-rank)**：业务策略调整
   - 去重：同作者间隔 ≥ 3
   - 多样性：类别打散
   - 新视频提权：冷启动流量扶持

### 2.3 评论排序算法

**热度 + 时间 + 身份 三因子加权：**

```
Score = 0.6 × hot_score + 0.3 × time_score + 0.1 × identity_boost

hot_score = log2(likes + 1) + 0.5 × log2(replies + 1)
time_score = exp(-0.693 × hours / 12)   // 半衰期 12 小时
identity_boost = 3.0 if 视频作者评论 else 0
```

### 2.4 直播系统架构

```
推流端(OBS/手机) --RTMP--> SRS流媒体服务器 --HTTP-FLV/HLS--> 观众端
                              |
                              v
                      ┌───────────────┐
                      │  Live Service  │
                      │  (Go + Redis)  │
                      └───────────────┘
                          │       │
              ┌───────────┘       └────────────┐
              v                                v
    ┌─────────────────┐              ┌──────────────────┐
    │  WebSocket Hub   │              │  Redis ZSET      │
    │  (弹幕/互动消息) │              │  (实时排行榜)     │
    │                  │              │                  │
    │  - 弹幕广播      │              │  ZADD room:rank  │
    │  - 礼物动画      │              │    user score     │
    │  - 点赞计数      │              │  ZREVRANGE top10  │
    │  - 系统通知      │              │                  │
    └─────────────────┘              └──────────────────┘
```

**排行榜 Redis 实现：**
```redis
# 送礼时更新分数
ZINCRBY live:rank:{room_id} {score} {user_id}

# 获取 Top 10
ZREVRANGE live:rank:{room_id} 0 9 WITHSCORES

# 直播结束持久化到 MySQL，清理 Redis
```

**点赞数据高并发方案：**
- 前端：合并请求，500ms 内的连续点赞合并成一次
- Redis：INCR 做实时计数，异步写 MySQL
- 防重：`SISMEMBER like:video:{id} {user_id}`
- 对账：定时任务比对 Redis 和 MySQL 数据

### 2.5 聊天系统设计

**WebSocket 长连接管理：**
```
Client <--WS--> Gateway <--Channel--> ChatService
                                        |
                    ┌───────────────────┼────────────────┐
                    v                   v                 v
                 MySQL              Redis              Kafka
              (消息持久化)      (在线状态/未读数)    (消息异步分发)
```

**消息可靠投递：**
1. 客户端发送消息，带 client_msg_id（客户端生成的UUID）
2. 服务端收到后生成 server_msg_id（Snowflake），写 Kafka
3. Kafka consumer 写 MySQL，更新未读计数
4. 通过 WebSocket 推送给在线接收方
5. 如果接收方不在线，下次拉取时通过 cursor 分页获取

**已读水位线：**
- 不逐条标记已读，而是记录最后读到的 msg_id
- `last_read_msg` 之前的全部算已读
- 减少写操作，适合高并发场景

### 2.6 数据库分库分表策略

**分片策略（亿级用户预案）：**

| 表 | 分片键 | 策略 | 分片数 |
|---|---|---|---|
| users | id | hash(id) % 256 | 256 |
| videos | id | hash(id) % 256 | 256 |
| video_likes | video_id | hash(video_id) % 64 | 64 |
| comments | video_id | hash(video_id) % 64 | 64 |
| messages | conversation_id | hash(conv_id) % 128 | 128 |
| user_behaviors | user_id | hash(user_id) % 32 + 按月分表 | 动态 |

**冷热分离：**
- 热数据：最近 30 天，SSD 存储
- 温数据：30-90 天，HDD 存储
- 冷数据：90 天以上，归档到对象存储

### 2.7 日志管理

**结构化日志 (Go zap)：**
```json
{
  "level": "info",
  "ts": "2026-04-13T15:30:01.234+0800",
  "caller": "handler/video.go:42",
  "msg": "video uploaded",
  "request_id": "req-abc123",
  "user_id": 10001,
  "video_id": 20001,
  "duration_ms": 234,
  "file_size": 15728640
}
```

**日志链路：**
```
App (zap JSON) --> Filebeat --> Kafka --> Logstash --> Elasticsearch
                                                          |
                                                      Kibana (查询/告警)
```

**日志级别策略：**
- `DEBUG`: 开发环境所有细节
- `INFO`: 关键业务操作（登录、上传、送礼）
- `WARN`: 可恢复异常（缓存miss、重试）
- `ERROR`: 不可恢复异常（DB连接失败、消息投递失败）

**RequestID 全链路追踪：**
- 每个请求生成唯一 request_id
- 在所有日志、RPC 调用中透传
- 出问题时一个 ID 串起整条链路

## 3. 部署架构（WSL 本地模拟）

```
WSL Ubuntu
├── Docker Compose
│   ├── MySQL 8.0          (3306)
│   ├── Redis 7            (6379)
│   ├── Kafka + Zookeeper  (9092)
│   ├── MinIO              (9000/9001)
│   ├── SRS                (1935/1985/8080→8088)
│   ├── Prometheus         (9090)
│   ├── Grafana            (3001)
│   └── ELK Stack          (5601/9200)
├── Backend Gateway         (8080)
└── Frontend Dev Server     (3000)
```

**一键启动：**
```bash
cd deploy/docker && docker compose up -d    # 基础设施
cd backend && go run ./cmd/gateway          # 后端
cd frontend && npm run dev                   # 前端
```
